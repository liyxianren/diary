CLAUDE.md 配置文件

什么是 CLAUDE.md？

CLAUDE.md 是 Claude Code 的长期记忆配置文件，放在项目根目录。每次启动 Claude Code 时，它会自动加载这个文件的内容作为上下文，让 AI 更好地理解你的项目。

为什么需要 CLAUDE.md？

🎯

提高准确性

告诉 AI 项目结构和规范，减少误操作

⚡

加快速度

AI 直接知道常用命令，无需反复询问

🔄

保持一致

确保编码风格和项目规范统一

📚

知识传承

团队成员都能快速了解项目

1

创建 CLAUDE.md 文件

在项目根目录创建文件：

touch CLAUDE.md
# 或在 Windows 中
echo. > CLAUDE.md
2

编写配置内容

以下是一个完整的 CLAUDE.md 模板示例：

# 项目：我的全栈应用

## 项目概述
这是一个使用 React + Node.js 构建的全栈 Web 应用。
前端使用 TypeScript，后端使用 Express 框架。

## 目录结构
```
├── frontend/          # React 前端代码
│   ├── src/
│   │   ├── components/  # 可复用组件
│   │   ├── pages/       # 页面组件
│   │   ├── hooks/       # 自定义 Hooks
│   │   └── utils/       # 工具函数
│   └── public/
├── backend/           # Node.js 后端代码
│   ├── src/
│   │   ├── routes/      # API 路由
│   │   ├── controllers/ # 业务逻辑
│   │   ├── models/      # 数据模型
│   │   └── middleware/  # 中间件
│   └── tests/
└── shared/            # 前后端共享代码
```

## 常用命令
- `npm run dev:frontend` - 启动前端开发服务器（端口 3000）
- `npm run dev:backend` - 启动后端开发服务器（端口 8080）
- `npm run build` - 构建生产版本
- `npm test` - 运行所有测试
- `npm run lint` - 代码检查
- `npm run format` - 代码格式化

## 编码规范
### TypeScript/JavaScript
- 使用 ES6+ 语法
- 优先使用函数式组件和 Hooks
- 使用 camelCase 命名变量和函数
- 使用 PascalCase 命名组件和类
- 文件名使用 kebab-case（如 user-profile.tsx）

### React 组件
- 每个组件一个文件
- 组件必须有 TypeScript 类型定义
- Props 接口命名为 ComponentNameProps
- 使用 ESLint 和 Prettier 保持代码整洁

### API 设计
- RESTful 风格
- 使用 HTTP 状态码：200（成功）、400（客户端错误）、500（服务器错误）
- 所有响应格式：`{ success: boolean, data?: any, error?: string }`

## 核心文件说明
- `frontend/src/App.tsx` - 前端主入口，路由配置
- `backend/src/server.ts` - 后端主入口，Express 配置
- `backend/src/config/database.ts` - 数据库连接配置
- `shared/types.ts` - 前后端共享类型定义

## 依赖说明
- 前端框架：React 18.x
- 状态管理：Redux Toolkit
- UI 组件库：Material-UI (MUI)
- HTTP 请求：Axios
- 后端框架：Express 4.x
- 数据库：PostgreSQL + Prisma ORM
- 认证：JWT

## 开发注意事项
1. 修改 API 接口后，记得同步更新 shared/types.ts 中的类型定义
2. 添加新的环境变量要在 .env.example 中说明
3. 数据库 schema 变更使用 `npx prisma migrate dev`
4. 提交前必须通过所有测试和 lint 检查
5. API 接口都有速率限制，测试时注意

## 测试规范
- 单元测试：使用 Jest
- 组件测试：使用 React Testing Library
- E2E 测试：使用 Playwright
- 测试文件命名：*.test.ts 或 *.spec.ts
- 测试覆盖率要求：>80%

rules 规则文件

什么是 rules 文件？

.claude/rules/*.md 是更细粒度的规则配置文件，可以创建多个规则文件，分别控制不同方面的行为。

CLAUDE.md vs rules

特性 CLAUDE.md rules 文件
位置 项目根目录 .claude/rules/
数量 一个 多个
用途 项目全局信息 特定领域规则
示例 项目结构、命令 代码规范、测试规则
1

创建 rules 目录

mkdir -p .claude/rules
2

创建规则文件示例

代码规范规则：.claude/rules/code-style.md

# 代码风格规则

## 命名规范
- 变量名使用 camelCase
- 常量使用 UPPER_SNAKE_CASE
- 类名使用 PascalCase
- 文件名使用 kebab-case

## 函数规范
- 每个函数必须有 JSDoc 注释
- 函数长度不超过 50 行
- 参数不超过 3 个，多了用对象传递

## 导入规范
- 按字母顺序排列导入语句
- 第三方库在前，本地模块在后
- 使用绝对路径导入（配置 tsconfig paths）

测试规则：.claude/rules/testing.md

# 测试规则

## 测试要求
- 每个功能必须有单元测试
- 关键业务逻辑必须有集成测试
- 测试覆盖率不低于 80%

## 测试命名
- 描述性命名：should...when...
- 例：should return error when user not found

## Mock 规范
- 使用 Jest mock 功能
- 外部 API 调用必须 mock
- 数据库操作必须使用测试数据库

提交规则：.claude/rules/git-commit.md

# Git 提交规则

## Commit Message 格式
```
(): 





```

## Type 类型
- feat: 新功能
- fix: 修复 bug
- docs: 文档更新
- style: 代码格式（不影响功能）
- refactor: 重构
- test: 测试相关
- chore: 构建或辅助工具变动

## 示例
```
feat(auth): add JWT token refresh mechanism

- Implement refresh token generation
- Add token rotation on refresh
- Update authentication middleware

Closes #123
```

💡 使用建议

  • 根据团队实际需求创建规则文件
  • 规则要具体明确，避免模糊表述
  • 定期更新规则，保持与项目同步
  • 新成员加入时，先阅读所有规则文件

Prompt 提示词精通

为什么 Prompt 如此重要？

Prompt（提示词）是你与 Claude Code 沟通的桥梁。好的 Prompt = 高质量的代码输出。掌握 Prompt 技巧，可以让 Claude Code 更准确地理解你的需求，生成更符合预期的代码。

Prompt 编写五大原则

1

具体明确

清楚说明要做什么，避免模糊表达

❌ 不好： 优化这个函数
✅ 好： 优化 calculateTotal 函数：1) 减少时间复杂度 2) 添加输入验证 3) 增加 JSDoc 注释
2

提供上下文

说明相关背景信息和约束条件

❌ 不好： 添加用户认证
✅ 好： 在 Express 后端添加 JWT 用户认证，需要兼容现有的 /api/auth 路由，使用 jsonwebtoken 库
3

分步骤说明

复杂任务分解为清晰的步骤

❌ 不好： 创建一个完整的用户管理系统
✅ 好： 创建用户管理系统：1) 设计数据库 schema 2) 创建 CRUD API 3) 添加权限验证 4) 编写单元测试
4

指定格式和风格

明确代码规范和输出格式

❌ 不好： 写个排序函数
✅ 好： 用 TypeScript 写一个泛型快速排序函数，使用箭头函数语法，添加完整的类型注解和 JSDoc
5

给出示例

提供期望的输入输出示例

❌ 不好： 创建日期格式化函数
✅ 好： 创建日期格式化函数，输入：Date 对象，输出："2025-01-15 14:30:00"格式的字符串

实用 Prompt 模板

模板1：创建新功能

创建 [功能名称]

要求：
1. [具体需求1]
2. [具体需求2]
3. [具体需求3]

技术栈：[列出使用的技术]
文件位置：[指定文件路径]
参考：[相关文件或代码]

测试要求：[是否需要测试，测试类型]

模板2：调试和修复

修复 [文件名] 中的 [问题描述]

错误现象：[详细描述错误]
错误位置：[文件路径:行号]
预期行为：[应该如何工作]
当前行为：[当前如何工作]

请：
1. 找出root cause
2. 修复问题
3. 添加防御性代码避免再次出现
4. 更新相关测试

模板3：重构优化

重构 [文件/函数名]

目标：
- [优化目标1，如性能提升]
- [优化目标2，如可读性]
- [优化目标3，如可维护性]

约束条件：
- 不改变外部 API 接口
- 保持向后兼容
- 所有现有测试必须通过

重构后需要：
1. 添加性能对比数据
2. 更新相关文档
3. 检查是否影响其他模块

模板4：代码审查

审查 [文件名] 的代码质量

关注点：
1. 是否符合项目编码规范
2. 潜在的性能问题
3. 安全隐患（如SQL注入、XSS等）
4. 边界条件处理
5. 错误处理是否完善
6. 代码可读性和可维护性

请给出：
1. 发现的问题列表
2. 改进建议
3. 优先级评级（高/中/低）

高级技巧

🎯 使用角色扮演

"作为一个资深的 React 开发者，帮我审查这个组件的性能问题"

🔄 迭代式对话

不要期望一次性完成，可以分步骤，每步验证后再继续

📋 引用已有代码

"参考 UserService.ts 的实现方式，创建 ProductService"

⚠️ 明确不要做什么

"修改配置，但不要改动生产环境的数据库连接"

模式切换（Alt+M）

Claude Code 的三种工作模式

按 Alt+M（Mac: Option+M）可以在不同模式间切换，每种模式适合不同的使用场景。

🤖

Auto 模式（自动编辑）

默认

特点

  • AI 自动查找文件、编辑代码、运行命令
  • 完全自主操作，无需人工确认
  • 适合快速开发和迭代

适用场景

  • ✅ 新功能开发
  • ✅ 代码重构
  • ✅ 批量修改
  • ✅ 日常开发任务

注意事项

⚠️ 重要文件建议先用 Plan 模式确认方案

📋

Plan 模式（规划模式）

按 Alt+M 切换

特点

  • AI 只制定计划，不执行代码修改
  • 用户审查计划后，决定是否执行
  • 降低风险，适合关键任务

适用场景

  • ✅ 大规模重构
  • ✅ 架构变更
  • ✅ 不熟悉的代码库
  • ✅ 生产环境代码

工作流程

  1. 切换到 Plan 模式（Alt+M）
  2. AI 分析任务，制定详细计划
  3. 用户审查计划，提出修改
  4. 确认后，AI 执行计划
✋

Manual 模式（手动模式）

按 Alt+M 切换

特点

  • AI 只提供代码建议，不自动修改
  • 用户手动复制粘贴代码
  • 完全掌控每一步操作

适用场景

  • ✅ 学习和研究
  • ✅ 代码审查
  • ✅ 敏感操作
  • ✅ 需要精细控制的场景

注意事项

💡 类似传统的 ChatGPT/豆包对话模式

模式对比速查表

功能 Auto 模式 Plan 模式 Manual 模式
自动编辑文件 ✅ 是 ❌ 否（需确认） ❌ 否
执行命令 ✅ 自动 ✅ 自动（确认后） ❌ 手动
风险控制 🟡 中等 🟢 低 🟢 最低
开发速度 🟢 最快 🟡 中等 🔴 较慢
学习成本 🟢 低 🟡 中 🟢 低

实际使用示例

场景 1：添加新功能（推荐 Auto 模式）

> 创建一个用户注册 API 接口

🤖 Auto 模式下：
1. [自动] 创建 routes/auth.ts
2. [自动] 创建 controllers/authController.ts
3. [自动] 更新 server.ts 注册路由
4. [自动] 创建测试文件
✅ 完成，无需任何确认

场景 2：重构核心模块（推荐 Plan 模式）

> 重构数据库连接模块，改用连接池

📋 Plan 模式下：
1. [显示计划]
   - 修改 database.ts，添加连接池配置
   - 更新所有调用数据库的文件
   - 调整错误处理逻辑
   - 更新测试用例
2. [等待确认] 您是否同意此方案？
3. [用户确认] 确认后开始执行
✅ 风险可控，每步都清楚

场景 3：学习代码（推荐 Manual 模式）

> 优化这个排序算法的性能

✋ Manual 模式下：
AI 返回优化后的代码和解释，但不会自动修改文件
你可以：
1. 仔细阅读代码和注释
2. 理解优化思路
3. 手动复制需要的部分
4. 自己调整和测试
✅ 适合学习和理解
